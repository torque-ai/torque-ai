# User Authentication System Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add username/password authentication with 4-role RBAC to TORQUE's dashboard, replacing API-key-only login while preserving API key auth for machine clients.

**Architecture:** New `user-manager.js` handles user CRUD with bcrypt hashing. New `role-guard.js` provides hierarchical role checking (admin > manager > operator > viewer). A centralized `isOpenMode()` replaces scattered `hasAnyKeys()` checks. The existing session/cookie infrastructure is reused — no new auth transport. Dashboard login form changes from API key input to username/password, with a setup wizard for first launch.

**Tech Stack:** Node.js, bcryptjs, better-sqlite3, Vitest, React

**Spec:** `docs/superpowers/specs/2026-03-21-user-auth-design.md`

- Password login returns 401 for wrong password

- Setup endpoint creates admin when no users exist, returns 201
- Setup endpoint returns 403 when users already exist
- Setup endpoint is rate-limited
- Auth status returns `needsSetup: true` when no users exist (but keys do)

- [ ] **Step 2: Update handleDashboardLogin to accept username/password**

In `server/api-server.core.js`, modify `handleDashboardLogin()` (line ~2053). **First**, update the open-mode check (line ~2073) from `!keyManager.hasAnyKeys()` to `isOpenMode()` — this is critical, otherwise a server with users but no keys would auto-login anyone. **Then** add a branch for `{ username, password }` before the existing `{ key }` branch. On success: validate via `userManager.validatePassword()`, create session, return `{ success, role, csrfToken, user: { id, username, displayName } }`. On failure: record rate limit, return 401.

**Security note:** In `validatePassword()`, always run `bcrypt.compare()` even when the user doesn't exist (compare against a dummy hash) to prevent username enumeration via timing differences.

- [ ] **Step 3: Add handleSetup handler**

New function: rate-limit check, verify `!userManager.hasAnyUsers()` (else 403), parse body, `createUser({ ..., role: 'admin' })`, create session, return 201 with user data.

**Security note:** Use a SQLite transaction or rely on the UNIQUE constraint on `username` to handle the TOCTOU race where two concurrent requests both pass the `hasAnyUsers()` check. The second INSERT will fail on the UNIQUE constraint, which the handler should catch and return 403.

- [ ] **Step 4: Add handleAuthStatus handler**

New function: check `isOpenMode()` → return `{ authenticated: true, mode: 'open', needsSetup: false }`. Check `!userManager.hasAnyUsers()` → `needsSetup: true`. Try `authenticate(req)` → return identity info or `{ authenticated: false, needsSetup }`. This endpoint has `skipAuth: true`.
