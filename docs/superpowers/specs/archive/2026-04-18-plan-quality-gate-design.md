# Plan Quality Gate — Design

## Goal

Reject vague auto-generated plans before the EXECUTE stage burns compute on
action-provider tasks that will fail verify and loop retrying. Today's
observed pattern: work items spend 10+ minutes running plan-task submissions
tagged `verify_retry=1` or `verify_retry=2` because the underlying plan
lacks concrete file paths, acceptance criteria, or clear scope. A scorer
that runs on each generated plan before submission rejects the obvious
junk, hands specific feedback back to the architect for one re-plan
attempt, and closes the item on the second failure.

## Placement

The gate runs **plan-level**, once per plan generation, inside
`executeNonPlanFileStage` in `server/factory/loop-controller.js`. It fires
after the plan markdown is written and after `lintAutoGeneratedPlan` passes,
and before the plan-executor starts submitting individual `## Task N:` blocks
to the action provider.

Rationale: today's failures are at the whole-plan level (the plan is
structurally vague), not task-N-of-M-looks-fine-but-task-K-is-bad. A single
gate integration is cheaper than per-task gating and matches the observed
failure pattern.

## Behavior summary

Two-pass scoring:

1. **Deterministic rules** (fast, zero LLM cost) — structural checks against
   the plan markdown. Produces a list of hard-fail violations and warnings.
2. **LLM semantic pass** (runs only when deterministic passes) — routes
   through the `plan_generation` category to a text-gen provider for a
   go/no-go + one-sentence critique. Catches "plan is concrete but solving
   the wrong problem".

Decision matrix:

| Trust level | Gate passes | Gate rejects (attempt 1) | Gate rejects (attempt 2) |
|---|---|---|---|
| supervised / guided | proceed to EXECUTE | pause at `PAUSED_AT_PLAN_REVIEW` with feedback | pause at `PAUSED_AT_PLAN_REVIEW` with feedback |
| autonomous / dark | proceed to EXECUTE | re-invoke plan-gen with feedback | reject work item, loop → IDLE |

Metadata override: `workItem.origin.skip_plan_quality_gate === true` bypasses
the gate entirely with an audit log entry. Research and open-ended exploratory
items can opt out.

Fail-open on gate errors: if the scorer itself throws, log a warn and treat
as pass. The gate must never wedge the factory.

## Deterministic rule set

Ten rules. Each is either a **hard fail** (rejects the plan) or a **warning**
(annotated, non-blocking).

### Structural

1. **`plan_has_task_heading`** — Plan contains at least one `## Task N:`
   heading. **Hard fail** if absent.
2. **`plan_task_count_upper_bound`** — Plan has ≤ 15 tasks. **Hard fail** if
   more. Over-scoped plans should be decomposed.
3. **`plan_task_count_lower_bound`** — Plan has ≥ 2 tasks. **Warning** if
   only one task (legitimate single-step items exist, but single-task plans
   are frequently a sign that the architect gave up).
4. **`task_body_min_length`** — Each task body (text between one `## Task N:`
   and the next) is ≥ 100 characters. **Hard fail** on any shorter.

### Per-task content

5. **`task_has_file_reference`** — Each task body mentions at least one
   concrete file path matching the regex
   `/[a-z_][a-z_0-9/]*\.(js|ts|tsx|jsx|py|cs|md|json|yml|yaml)/i` OR a grep
   target (`search_files`, explicit symbol name invocation). **Hard fail** on
   any task missing both.
6. **`task_has_acceptance_criterion`** — Each task body contains at least one
   of: a named test file, a runnable command (`npx vitest`, `dotnet test`,
   `pytest`), a file path that should exist after the change, or an explicit
   `assert`/`expect` line. **Hard fail** on any task missing all.
7. **`task_avoids_vague_phrases`** — No single task body contains ≥ 2
   occurrences of vague phrases from the list: `appropriately`, `as needed`,
   `refactor accordingly`, `clean up`, `improve`, `fix issues` (without
   further context), `etc.` as a list terminator. **Hard fail** on any task
   with ≥ 2 hits. The 2-hit threshold reduces false positives for legitimate
   single uses.

### Plan shape

8. **`no_duplicate_task_titles`** — No two `## Task N:` headings share the
   same title text (after trimming `N:` and whitespace). **Hard fail** on
   duplicates.
9. **`task_heading_grammar`** — Plan uses `## Task N:` grammar, not
   `## Task 0:`, `## Step 1:`, or other variants. **Hard fail** on mismatch.

### Budget

10. **`plan_size_upper_bound`** — Total plan markdown ≤ 100 KB. **Hard fail**
    if larger. Runaway plans should be decomposed.

### Known limitations (follow-ups, not blockers)

- Rule 5's regex doesn't match file paths with unicode characters.
- Rule 7 runs on raw markdown; forbidden phrases inside code fences trigger
  as normal text. Mitigated by the ≥ 2-hits-per-task threshold.

## LLM semantic pass

Runs only when the deterministic pass produces zero hard fails. Uses the
existing `plan_generation` routing category (cerebras / groq / ollama in
default templates).

- Input: work item title + description + generated plan markdown
- Prompt: asks for a go / no-go verdict plus a single-sentence critique
  explaining the verdict
- Token budget: under 5 KB prompt, under 200 tokens output
- Timeout: 60 seconds, no retry
- Failure modes: timeout / provider error / unparseable output → return null
  critique, treat as pass

The LLM is a tiebreaker for "plan is structurally fine but semantically
misses the point". It cannot reject a plan that passed deterministic unless
it returns an explicit no-go verdict.

## Components

### New file: `server/factory/plan-quality-gate.js`

Main entry:

```js
async function evaluatePlan({ plan, workItem, project, options = {} })
  → {
      passed: boolean,
      hardFails: [{ rule: string, taskNumber?: number, detail: string }],
      warnings: [{ rule: string, taskNumber?: number, detail: string }],
      llmCritique: string | null,
      feedbackPrompt: string | null,
    }
```

Internal helpers, exported for tests:

```js
function runDeterministicRules(planMarkdown)
  → { hardFails, warnings }

async function runLlmSemanticCheck({ plan, workItem, timeoutMs })
  → string | null   // critique text or null on failure

function buildFeedbackPrompt(hardFails, warnings, llmCritique)
  → string          // the "## Prior plan rejected — address these issues" block
                    // returns null when passed
```

Constants:

```js
const RULES = { /* rule metadata keyed by id */ };
const MAX_REPLAN_ATTEMPTS = 1;
const LLM_TIMEOUT_MS = 60_000;
```

### Modification: `buildAutoGeneratedPlanPrompt`

Extend signature to accept an optional third argument:

```js
function buildAutoGeneratedPlanPrompt(project, targetItem, priorFeedback = null)
```

When `priorFeedback` is a non-empty string, prepend a clearly labeled section
`## Prior plan rejected — address these issues` to the prompt before the
existing content.

### Modification: `executeNonPlanFileStage` in `loop-controller.js`

After existing `lintAutoGeneratedPlan` success:

1. Read `plan_gen_attempts` from `workItem.origin?.plan_gen_attempts`
   (default 0).
2. Check `workItem.origin?.skip_plan_quality_gate`. If true: emit
   `factory:plan_gate_skipped`, log decision, proceed to EXECUTE.
3. Otherwise: call `evaluatePlan({ plan, workItem, project })`.
4. On pass: log `plan_quality_passed`, proceed to EXECUTE.
5. On fail with `attempts === 0`:
   - supervised/guided → pause at `PAUSED_AT_PLAN_REVIEW` with
     `stage_result.gate_feedback` populated
   - autonomous/dark → update `origin_json` with `plan_gen_attempts: 1` and
     `last_gate_feedback`, re-invoke `submitFactoryInternalTask` with the
     feedback prompt, re-run `lintAutoGeneratedPlan`, re-run `evaluatePlan`
6. On fail with `attempts === 1`:
   - supervised/guided → same pause path (operator decides)
   - autonomous/dark → mark item `status='rejected'` with
     `reject_reason='plan_quality_gate_rejected_after_2_attempts'`,
     return `{ stop_execution: true, next_state: 'IDLE' }`

### Event-bus additions: `server/event-bus.js`

```js
emitFactoryPlanRejectedQuality({ project_id, work_item_id, rule_violations, attempt })
emitFactoryPlanRejectedFinal({ project_id, work_item_id, rule_violations })
emitFactoryPlanGateSkipped({ project_id, work_item_id, reason })
```

### Decision-log actions

New `safeLogDecision` actions in the existing pattern:

- `plan_quality_passed`
- `plan_quality_rejected_will_replan`
- `plan_quality_rejected_final`
- `plan_quality_skipped_by_metadata`
- `plan_quality_gate_fail_open` (on scorer exception)

## Data model

No new DB tables or columns. The gate persists its state via
`factory_work_items.origin_json`, which is already a free-form JSON blob:

```json
{
  "plan_path": "...",
  "plan_gen_attempts": 1,
  "last_gate_feedback": {
    "hardFails": [ ... ],
    "warnings": [ ... ],
    "llmCritique": "..."
  }
}
```

Plan markdown files are NOT deleted on rejection — they stay at the
auto-generated path for forensic inspection. `plan_gen_attempts` is the
source of truth for attempt counting.

## Error handling

### Gate throws

```js
try {
  const verdict = await evaluatePlan({ plan, workItem, project });
  // ...
} catch (err) {
  logger.warn('plan-quality-gate evaluation failed; treating as pass', {
    project_id: project.id,
    work_item_id: workItem.id,
    err: err.message,
  });
  safeLogDecision({ action: 'plan_quality_gate_fail_open', ... });
  // proceed to EXECUTE
}
```

### LLM pass errors

- Timeout → null critique, deterministic result stands
- Provider error (4xx/5xx, no key configured) → null critique
- Unparseable LLM output → treat as go, deterministic result stands
- Garbage input (plan is bytes of gibberish) → rule 1 catches before LLM
  pass runs

### Re-plan generation errors

If the re-plan `submitFactoryInternalTask` itself fails (provider outage,
task never completes):

```js
if (generationTask.status !== 'completed') {
  safeLogDecision({ action: 'plan_quality_replan_generation_failed', ... });
  factoryIntake.updateWorkItem(id, {
    status: 'rejected',
    reject_reason: 'replan_generation_failed',
  });
  return { stop_execution: true, next_state: 'IDLE' };
}
```

### Concurrency

The gate is stateless (no in-memory caches). Two projects running gate
evaluation concurrently don't share state. The gate runs inside a single
instance's `executeNonPlanFileStage` call, which is already serialized per
instance by the loop machinery.

## Testing

### Unit tests: `server/tests/plan-quality-gate.test.js`

Each of the 10 rules gets positive + negative coverage. Representative
subset:

- Rule 1: pass with valid plan, hard fail on empty string, hard fail on
  prose-only
- Rule 3: pass with 2+ tasks, **warning** on 1
- Rule 5: pass on `src/foo.ts`, pass on `search_files: "handleFoo"`, hard
  fail on task with no file or grep reference
- Rule 6: pass on `npx vitest` mention, pass on `assert` line, hard fail on
  task with neither
- Rule 7: pass on single "appropriately", hard fail on two forbidden phrases
  in the same task
- Rule 10: pass at 99 KB, hard fail at 101 KB

LLM-pass gating:

- Deterministic hard fail → `runLlmSemanticCheck` not called (mock, assert 0
  calls)
- Deterministic pass → `runLlmSemanticCheck` called, returns critique,
  verdict respects it
- Deterministic pass + LLM timeout → treated as pass, critique null
- Deterministic pass + LLM returns garbage → treated as pass, critique is
  the raw string

Feedback builder:

- Mixed hard fails + warnings + LLM critique → structured violations section
  plus prose semantic concern section
- Warnings only → null (non-blocking)
- Empty → null

### Integration test: `server/tests/factory-plan-quality-gate-e2e.test.js`

Full flow through `executeNonPlanFileStage`:

- Scenario 1: autonomous, good plan first try → `plan_gen_attempts=1`,
  EXECUTE fires
- Scenario 2: autonomous, bad plan 1st, good plan 2nd →
  `plan_gen_attempts=2`, EXECUTE fires
- Scenario 3: autonomous, bad plan twice → `status='rejected'`,
  `reject_reason='plan_quality_gate_rejected_after_2_attempts'`, loop →
  IDLE
- Scenario 4: supervised, bad plan → `paused_at_stage='PLAN_REVIEW'`,
  `stage_result.gate_feedback` populated
- Scenario 5: `skip_plan_quality_gate=true` → `evaluatePlan` not called,
  EXECUTE fires
- Scenario 6: stub `evaluatePlan` to throw → factory proceeds to EXECUTE,
  decision log has `plan_quality_gate_fail_open`

Use `server/tests/vitest-setup.js` helpers. Stub `submitFactoryInternalTask`
via require-cache replacement. Stub the LLM pass via
`vi.spyOn(planQualityGate, 'runLlmSemanticCheck')`.

### Lint coverage

Under `server/factory/`, today's lint rules apply:

- `no-hardcoded-factory-provider` — the LLM pass must use smart routing, not
  hardcoded provider
- `no-spawn-sync-in-factory` — no spawnSync anywhere in the module

## Out of scope

- Learning thresholds from historical verify-pass/verify-fail data
- Per-project rule tuning (all projects share the same rules initially)
- A dashboard view of gate rejections (surfaced via existing event-bus, but
  no dedicated UI)
- Rule 5 unicode-path support
- Rule 7 code-fence-aware parsing

These are candidates for a v2 once the v1 telemetry (rejection event rates,
false-positive reports) tells us what actually matters.
