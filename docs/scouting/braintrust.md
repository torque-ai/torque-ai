# Findings: Braintrust

**Tagline:** SDK-first eval platform that turns code, logs, playgrounds, and gateway traffic into comparable experiments.
**Stars:** 392 (GitHub, 2026-04-12)
**Language:** TypeScript

## Feature 1: `Eval()` as the Primary Interface
**What it does:** Braintrust centers evaluation around a compact SDK pattern: `Eval(name, { data, task, scores })` in TypeScript and equivalent shapes in Python, Go, Ruby, Java, and C#. Running that code locally or in CI creates an immutable experiment, prints summary metrics in the terminal, and links the run into the Braintrust UI. It also supports editable `parameters`, so model choice, prompts, and temperatures can be exposed to the playground without rewriting the eval structure.
**Why distinctive:** Compared with Promptfoo's config-first matrix and DeepEval's Python-native test case objects, Braintrust makes the core eval abstraction unusually small and direct. The interesting part is not just brevity, but that the exact same code shape already maps to the platform's experiment model, so the UI feels like a view on top of code rather than a separate eval system.
**TORQUE relevance:** HIGH - Plan 70 would benefit from a minimal `Eval(name, data, task, scores)` contract instead of a large custom config surface. TORQUE is already code- and workflow-driven, so Braintrust's SDK-first pattern is a better fit than a dashboard-first eval product and would make it easier to wrap provider calls, workflow steps, or verifier logic into standard experiments.

## Feature 2: Scorers That Travel Across Offline and Online Evaluation
**What it does:** Braintrust treats scorers as first-class reusable components: prebuilt autoevals, LLM-as-a-judge scorers, and custom code scorers. They can be defined inline, created in the UI, or pushed via CLI, and custom code scorers can evaluate either individual spans or whole traces. The same scorer model is used for offline experiments, playground testing, and online scoring of live production traces.
**Why distinctive:** The distinctive part is portability, not just metric variety. Promptfoo has a wider assertion catalog and DeepEval has strong metric classes, but Braintrust is cleaner about making one scorer artifact move between dev-time evals and production monitoring without changing the surrounding workflow.
**TORQUE relevance:** HIGH - TORQUE needs both offline quality checks and live quality monitoring across Plans 70 and 68. Braintrust's scorer model suggests a way to avoid separate "eval assertions" and "runtime monitors" by letting one evaluator object run against benchmark cases, traces, and promoted production failures.

## Feature 3: Immutable Experiments with Built-In Diff Mode
**What it does:** Braintrust stores each experiment as an immutable snapshot of an evaluation run. When multiple experiments use the same dataset, the UI automatically compares them, highlights per-row score deltas, marks improvements in green, marks regressions in red, and lets you choose a baseline in diff mode. It also supports hill-climbing flows where a previous experiment's output can become the next run's comparison target.
**Why distinctive:** A lot of eval tools can rerun suites, but Braintrust is especially opinionated about experiments as durable, diffable artifacts. Compared with Promptfoo and DeepEval, it leans harder into regression analysis inside the experiment viewer itself instead of leaving before/after comparison mostly to external reporting or manual interpretation.
**TORQUE relevance:** HIGH - TORQUE needs explicit before/after evidence when prompts, providers, or agent policies change. Immutable experiments plus a first-class diff view would give Plan 70 a much better regression surface than raw logs, one-off benchmark scripts, or manually compared score tables.

## Feature 4: Versioned Datasets Fed by Production Data
**What it does:** Braintrust datasets are integrated with the rest of the platform and version every insert, update, and delete, so evaluations can pin to an exact dataset version through the SDK. Datasets can pull examples from production logs, staging runs, manual review, and previous evaluations, and Braintrust explicitly states that logs use the same data structure as experiments. That means production traces, human feedback, and eval runs all sit on one compatible record model.
**Why distinctive:** This is more than dataset storage. Compared with Promptfoo's external fixture files and DeepEval's more code-local test assets, Braintrust is stronger at turning real traffic into reproducible benchmark material without changing schemas or leaving the platform boundary.
**TORQUE relevance:** HIGH - This is where Plan 68 observability and Plan 70 evals meet. TORQUE could promote interesting workflow traces into versioned eval corpora, rerun changes against the exact same cases, and keep feedback, telemetry, and benchmark inputs on one shared structure instead of splitting them across separate systems.

## Feature 5: Playground Iteration Plus Gateway/Proxy as a Byproduct
**What it does:** Braintrust uses playgrounds as mutable iteration spaces where teams can test prompts and models side by side, tune scorers, and then promote winning configurations into immutable experiments. On the runtime side, the Braintrust gateway exposes an OpenAI-compatible base URL that routes requests across providers while automatically logging them to a Braintrust project; the open-source `braintrust-proxy` repository shows the same unified-API and caching pattern, although the docs now mark the proxy as deprecated in favor of the gateway.
**Why distinctive:** The important design is that the gateway is not a separate add-on product with a disconnected data model. Braintrust makes experimentation, tracing, prompt iteration, and model access converge on the same platform loop, so the gateway/proxy feels like an operational consequence of the eval system rather than an unrelated API layer.
**TORQUE relevance:** HIGH - TORQUE already has provider routing and wants stronger observability. A gateway-style layer that emits the same artifacts used for evaluation would connect Plan 68 and Plan 70 cleanly: runtime traffic becomes traces, traces become datasets, and datasets feed the same experiment and diff workflow used during development.

## Verdict
Braintrust's strongest idea is not any single dashboard feature, but the way `Eval()`, scorers, datasets, diffable experiments, playgrounds, and gateway traffic all collapse into one SDK-first loop. That makes it a particularly strong reference for TORQUE because Plans 68 and 70 currently read like separate observability and eval tracks, while Braintrust shows a cleaner shared model. The most portable ideas are the minimal `Eval(name, data, task, scores)` contract, reusable scorers that run both offline and online, and immutable diffable experiments backed by versioned datasets.
